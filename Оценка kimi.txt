Вот честный разбор — что реально осталось не доделанным и как это закрыть. Проект сильно вырос (8.5/10), но есть 3 категории остатков: баги-на-будущее, не закрытые фичи из промтов и инфраструктурный долг.
🔴 Что не так (остаточные проблемы)
1. React Hooks — 28 warnings ESLint (потенциальные баги)
react-hooks/exhaustive-deps и set-state-in-effect — это не просто "warning'и в консоли", это race conditions и infinite loops в продакшене. В корпоративном ПО, где сессии длинные, это критично.
Где искать: скорее всего в Dashboard.tsx, Tickets.tsx, TicketDetail.tsx, ChatDetail.tsx — там где useEffect + setState без guard'ов, и хуки с пропущенными deps.
2. Админ-панель — не закрытые фичи из промта #10
Нет настроек компании (название, логотип, часовой пояс)
Нет email-шаблонов с переменными ({{userName}}, {{ticketId}})
Audit log пишется вручную (auditLogMiddleware вставлен в 12 файлов), но нет автоматического middleware на ВСЕ POST/PUT/DELETE
Нет CSV экспорта аудита
Нет фильтров по дате/типу в аудите
3. Уведомления — не закрытые фичи из промта #11
Нет уведомлений из чатов и календаря (только тикеты)
Нет страницы "Все уведомления" (только dropdown колокольчика)
Нет read receipts в чатах (WebSocket)
4. WebSocket — не закрытые фичи из промта #12
Нет user:online/offline broadcast (есть last_active в БД, но не real-time статус)
Нет offline message queue (если юзер не в сети — сообщения теряются для WebSocket, остаются только в БД)
Нет комнат ticket:{id} для real-time обновлений тикета (есть REST polling через React Query)
5. Серверное покрытие — 64% → цель 70%
background.js — нет тестов на BullMQ fallback (setInterval)
search.js — нет тестов на FULLTEXT → LIKE fallback и SQL-injection guard
tickets.service.js — 85%, остались ветки 500-ответов при throw
6. Интеграционные тесты
check-console.mjs проверяет 6 страниц, нужно 12+ (/admin, /calendar, /polls, /wiki, /chats, /employees, /search, /files, /profile, /kanban, /notifications, /register)
Playwright 22 теста — мало для production. Нет сценариев: LDAP-логин, загрузка файла + ClamAV, SLA эскалация, drag&drop в чатах
7. Инфраструктура
Helmet CSP может ломать Vite HMR в dev — нужно conditional policy
WebSocket rate-limit — 10 handshake/min per IP ломается за NAT (100 человек в офисе = 1 IP). Нужен whitelist внутренних сетей или skipSuccessfulRequests
Sentry — не интегрирован в error-handling middleware (PII filter есть, но не подключен к window.onerror)

Этап E (1-2 дня) — WebSocket: online/offline + ticket rooms
Что делать:
Online/offline — в socket.js:
При connection → io.emit('user:online', { userId })
При disconnect → io.emit('user:offline', { userId })
Обновлять isOnline в Prisma (или использовать last_active + 5 минут логику на фронте)
Ticket rooms — при входе в TicketDetail.tsx:
TypeScript
socket.emit('ticket:join', ticketId);
Сервер: socket.join(ticket:${ticketId})
При изменении статуса/приоритета тикета: io.to(ticket:${id}).emit('ticket:updated', ticket)
На фронте: слушать ticket:updated и делать queryClient.invalidateQueries(['ticket', id])
Offline message queue — опционально, можно skip если используется React Query (данные и так из БД приходят). Но для "push-уведомлений" — нужно накапливать в Redis offline:{userId} при отсутствии сокета.

WS rate-limit NAT fix — skipSuccessfulRequests: true + whitelist local subnets:
JavaScript
const wsRateLimit = new RateLimiter({
  keyPrefix: 'ws_handshake',
  points: 10,
  duration: 60,
  skip: (req) => ['127.0.0.1', '10.0.0.0/8', '192.168.0.0/16'].includes(req.ip)
});
