2. Небезопасная работа с токеном в logout
TypeScript
// src/context/AuthContext.tsx — logout
fetch(`${API_URL}/auth/logout`, {
  method: 'POST',
  credentials: 'include',
  headers: { Authorization: `Bearer ${token}` }, // token уже null!
})
token обнулён через setToken(null) до fetch. Запрос уйдёт без авторизации, refresh token не инвалидируется на сервере.
Исправление:
TypeScript
const logout = () => {
  const t = token // сохраняем до очистки
  localStorage.removeItem('token')
  localStorage.removeItem('user')
  setToken(null)
  setUser(null)
  if (t) {
    fetch(`${API_URL}/auth/logout`, {
      method: 'POST',
      credentials: 'include',
      headers: { Authorization: `Bearer ${t}` },
    }).catch(() => {})
  }
}
3. Race condition в tickets.js — raw SQL без параметризации
JavaScript
// server/src/routes/admin.js
const rows = await prisma.$queryRaw`
  SELECT ... FROM employees ORDER BY is_active DESC, name
`
Хотя Prisma экранирует $queryRaw с template literal, это ломает типобезопасность и переносимость. Лучше prisma.employees.findMany().
4. Отсутствует requireRole('super_admin') на повышение роли
JavaScript
// admin.js — PUT /users/:id
if (role && ['admin', 'senior_agent', 'agent', 'requester'].includes(role)) {
  data.role = role
}
super_admin нет в списке, но и запрет повышения до super_admin через API проверяется только в одном месте (Этап 16). Если req.user.role === 'super_admin', он может назначить super_admin? Нет, потому что 'super_admin' не проходит includes(). Но это неявно — лучше явный guard:
JavaScript
if (role) {
  if (role === 'super_admin') {
    return res.status(403).json({ message: 'Cannot assign super_admin via API' })
  }
  if (!['admin', 'senior_agent', 'agent', 'requester'].includes(role)) {
    return res.status(400).json({ message: 'Invalid role' })
  }
  data.role = role
}
🟡 Средние проблемы
5. Дублирование API префиксов
JavaScript
// app.js
function mount(prefix, router, ...mw) {
  app.use(`/api${prefix}`, ...mw, router)
  app.use(`${V1}${prefix}`, ...mw, router) // /api/v1/tickets
}
Итог: /api/tickets и /api/v1/tickets — дубль. В production это раздувает поверхность атаки и путает кэш. Лучше оставить один префикс.
6. Cache middleware ломает SSE/WebSocket upgrade
JavaScript
// cache.js — cacheMiddleware
res.json = (body) => {
  cache.set(key, body, ttl)
  originalJson(body)
}
Если route использует res.write() или streaming — кэш не сработает, но и не сломает. Однако res.json переопределён навсегда для запроса. Если middleware вызван несколько раз (чего не бывает в Express, но всё же) — будет рекурсия.
7. Missing error boundary на lazy-loaded админках
tsx
// App.tsx
const Admin = lazy(() => import('@/pages/Admin'))
// ...
<Route path="/admin" element={<ProtectedRoute adminOnly><AdminLayout /></ProtectedRoute>}>
AdminLayout не обёрнут в ErrorBoundary, в отличие от остальных страниц. Если админка упадёт — краш всего приложения.
8. Zod v4 в package.json, но schemas используют v3 API
TypeScript
// src/lib/schemas.ts
import { z } from 'zod'
z.string().email('...')
Zod v4 изменил API (z.string().email() без аргументов, сообщения через .message()). Нужно либо понизить до ^3.x, либо мигрировать схемы.
🟢 Мелочи и полировка
9. i18n — fallbackLng: "ru", но детектор может выбрать "en"
TypeScript
// i18n/index.ts
fallbackLng: "ru",
detection: { order: ["localStorage", "navigator"], caches: ["localStorage"] },
Если у пользователя браузер на английском и чистый localStorage — загрузится en. Это ок, но для корп. системы лучше форсировать ru по умолчанию:
TypeScript
fallbackLng: "ru",
lng: localStorage.getItem('i18nextLng') || 'ru',
