Что сделано — Service Desk (Корпоративная система тикетов)
=========================================================

Стек: React 18 + TypeScript + Vite + Tailwind CSS + shadcn/ui + Express + MySQL + Socket.io

────────────────────────────────────────────────────────────────────────────

МОДУЛИ

  ✅ Аутентификация и RBAC
    • JWT-логин / регистрация / dev-login
    • Три роли: admin, senior_agent, agent
    • requireRole() middleware на бэкенде
    • express-validator на 7 роутах
    • ProtectedRoute + скрытие UI элементов по роли
    • Восстановление пароля (nodemailer)

  ✅ Тикеты
    • CRUD, статусы (open → in_progress → resolved → closed)
    • Приоритеты (low / medium / high / critical)
    • Назначение на сотрудника
    • Чат внутри тикета + внутренние заметки
    • Вложения (multer → uploads/tickets/)
    • WebSocket-эмиты изменений в реальном времени
    • CSV (UTF-8 BOM) и PDF (jsPDF + html2canvas) экспорт

  ✅ Дашборд
    • Статистика: всего, активные, критические, решённые
    • График по статусам (Recharts BarChart)
    • Виджет сотрудников (онлайн/офлайн)
    • Последние обновления тикетов
    • Клик по карточкам → переход с фильтром

  ✅ Чаты (WebSocket)
    • Общие и личные чаты
    • Отправка/удаление сообщений в реальном времени
    • Поиск по чатам и внутри чата
    • Реакции (эмодзи)
    • Непрочитанные сообщения (бейдж)

  ✅ Wiki
    • Статьи, категории, поиск
    • Загрузка изображений (multer → uploads/wiki/)
    • Создание через диалог

  ✅ Новости
    • Лента новостей с фильтром «Важные»
    • Создание новостей (senior_agent+)

  ✅ Календарь
    • Сетка с навигацией по месяцам
    • Создание / удаление событий
    • Ближайшие события (виджет)

  ✅ Опросы
    • Список, создание, голосование
    • Прогресс-бары результатов

  ✅ Файлы
    • Папки, drag & drop, загрузка (multer → uploads/files/)
    • Фильтр по типу, поиск, вид сетка/список
    • Открытие файлов по клику

  ✅ Сотрудники
    • Список с онлайн-статусом
    • Поиск, фильтр по роли/отделу
    • Карточки / таблица (переключение)

  ✅ Глобальный поиск
    • LIKE-поиск по 6 таблицам: тикеты, сотрудники, wiki, новости, чаты, файлы
    • Ctrl+K / боковое меню
    • Группировка результатов по разделам

  ✅ Админ-панель
    • Дашборд, управление пользователями (роли, блокировка)
    • Push-уведомления (подписка / отправка)
    • Настройки интеграций (email, телеграм)
    • Аудит действий

  ✅ WebSocket (Socket.io)
    • JWT-аутентификация на handshake
    • Комнаты chat:{id}, эмиты message:new / message:removed
    • Эмиты ticket:created / ticket:updated / ticket:message
    ● Redis adapter при REDIS_URL (иначе in-memory)

ЗАПУСК

  Требования: Node.js 18+, MySQL (Laragon / Docker)

  Быстрый старт (локально):
    cp server\.env.example server\.env
    cd server && npm run dev
    cd .. && npm run dev
    → http://localhost:5173 (фронтенд)
    → http://localhost:4000/api (API)
    → http://localhost:4000/api/docs (Swagger)

  Docker (prod):
    docker compose up -d --build
    → http://localhost

  E2E тесты:
    npx playwright install chromium
    npm run test:e2e

  DevOps логин: POST /api/auth/dev-login (без пароля, пользователь id=1)

АУДИТ ПРОЕКТА (2026-07)

  Проверка по типовому чек-листу корпоративного Service Desk — сделано 7
  специализированных поисков по коду параллельно.

  Безопасность:
    ✅ bcrypt (10 раундов), параметризованные SQL-запросы, rate-limit,
       helmet, express-validator (7 роутов)
    🔴 /uploads — express.static БЕЗ аутентификации (любой файл по URL)
    🔴 /dev-login не отключается в production (выдаёт admin без пароля)
    🟡 /api/system-info без auth (раскрывает имя ПК/домен)

  Тестирование:
    ✅ 14 клиентских + 17 серверных тестов, Playwright + Chromium
    🔴 Нет отдельной тестовой БД (тесты ходят в реальную servicedesk)
    🔴 CI не гоняет клиентские тесты и не поднимает MySQL
    🟡 Клиентские тесты — 3/20 страниц, без MSW, E2E только Login

  Обработка ошибок:
    ✅ Глобальный error handler на сервере, console.error во всех catch
    🔴 Нет Error Boundary на клиенте (render crash → белый экран)
    🔴 Нет единого API-интерцептора (API в 16 местах, ошибки глотаются)

  UI / a11y / i18n:
    ✅ Нижняя мобильная навигация, тёмная тема, i18n-файлы полные
    🔴 i18n не используется в 80% страниц (весь UI хардкод на русском)
    🔴 Ноль aria-label, label без htmlFor/id, кликабельные div без keyboard
    🟡 Tabs — кастом без Radix, Drag-and-drop без клавиатуры

  Функциональные дыры:
    ✅ Email, Telegram, WebPush, WebSocket, аудит, PDF/CSV на тикетах
    🟡 Нет LDAP/AD/SSO — аутентификация только email+password
    🟡 Экспорт только на странице тикетов (нет для Wiki/News/Calendar/etc)

  Производительность:
    ✅ Redis для Socket.io (опционально, fallback in-memory)
    🔴 Нет серверной пагинации на 9/12 эндпоинтов (возвращают всё)
    🔴 Нет ни одного CREATE INDEX (только PK + 2 UNIQUE)
    🟡 N+1 в polls.js и files.js, client-side пагинация

  Инфраструктура:
    ✅ Docker, K8s, Vercel, Tauri, CI/CD, Swagger, Playwright
    🔴 Server Dockerfile не multi-stage, нет non-root USER
    🔴 JWT_SECRET по умолчанию в compose, нет server/.dockerignore

ПЛАН ДОРАБОТКИ ПО ЭТАПАМ

  Этап 13 — Безопасность (P0)
    • Отключить /dev-login в production (NODE_ENV !== 'production')
    • Закрыть /uploads JWT-аутентификацией (middleware на статику)
    • Client Error Boundary вокруг <App>
    • Убрать дублирование { error } / { message } в auth.js (2 места)
    • /api/system-info — либо убрать, либо защитить auth

  Этап 14 — Производительность (P1)
    • Серверная пагинация LIMIT/OFFSET на tickets, news, wiki, files, polls
    • CREATE INDEX: tickets(status, created_at, assigned_to, created_by),
      ticket_messages(ticket_id, created_at), files(folder_id),
      events(date), notifications(user_id, created_at)
    • Исправить N+1 в polls.js (Eager load options) и files.js (JOIN вместо 2 запросов)
    • Client-side → server-side пагинация на Tickets и News (фронт под параметры)

  Этап 15 — Инфраструктура (P1)
    • Docker: multi-stage для server, non-root USER, server/.dockerignore
    • CI: добавить npm test для клиента, MySQL service для серверных тестов
    • Отдельная тестовая БД (DB_NAME=servicedesk_test, поднимается в CI)
    • VITE_API_URL как ARG в Dockerfile frontend

  Этап 16 — UI / i18n / a11y (P2)
    • i18n — пройти по всем страницам, заменить хардкод на t()
    • aria-label на иконки (Menu, Logout, Sort, View toggle)
    • htmlFor/id на все label + input (Login, Register, NewTicket, Files)
    • keyboard + role="button" + tabIndex на кликабельные карточки
    • Error messages с role="alert"
    • Tabs — Radix или aria-атрибуты вручную
    • Drag-and-drop клавиатурная альтернатива (Files, Kanban)

  Этап 17 — Опционально (P3)
    • LDAP/AD интеграция (passport-ldapauth)
    • Redis-кэш для API-ответов (GET /api/tickets, /api/employees)
    • Экспорт данных с других страниц (Wiki, Calendar, Employees, News)
    • MSW на клиенте для тестов страниц с API
    • Покрытие тестами 80% (c8), E2E: ticket CRUD, чаты, админка
    • WebSocket rate-limit (5 msg/sec) + exponential backoff
    • Очистка уведомлений > 90 дней
    • Лимит экспорта 10 000 записей
